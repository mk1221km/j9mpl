### NetRexx Grammar Rules & Invariants:
1. Imports must be nominal (no wildcards):
   import java.sql.DriverManager
   import java.sql.Connection
   import java.sql.Statement
   import java.sql.PreparedStatement
   import java.sql.ResultSet
   import java.sql.SQLException
2. Method Signatures:
   Always declare arguments with an equals sign (=), e.g., `method initRoutingTable(dbPath = String) public static`.
   DO NOT omit the equals sign in method arguments.
   Specify the return type with `returns <Type>`, e.g., `method getTransactionCount(dbPath = String, status = String) public static returns Rexx`.
3. Line Continuation:
   If a statement or string concatenation is split across lines, you MUST terminate the line with a trailing hyphen (`-`) to continue the statement, e.g.:
   stmt.executeUpdate("CREATE TABLE IF NOT EXISTS routing_rules (" || -
     "min_amount REAL, " || -
     "priority TEXT)")
   Without a trailing hyphen `-` at the end of the line, NetRexx treats the newline as the end of the statement, causing compilation errors.
4. Exception Handling & DO Blocks:
   Use do/catch/finally/end blocks. Every single `do` block (including `do`, `if ... then do`, `else do`, `loop while`, and `finally do`) MUST have exactly one matching `end` keyword. Do NOT generate extra or mismatched `end` keywords. Mismatched do/end counts cause compilation errors.
5. Database Path Safety Guard:
   To prevent SQLite from physically creating a file named 'null' in the current working directory, you MUST guard all database connection code with `if dbPath \= null & dbPath \= "null" then do` before establishing a connection.
6. Void Methods (Returning Nothing):
   In NetRexx, methods that do not return a value (void methods) MUST NOT have a `returns` clause. Do NOT write `returns void` or `returns none`. Simply omit the `returns` keyword and type completely. Example: `method logMetric(dbPath = String, record = MetricRecord) public static`.
7. No Terminating 'end' for Methods:
   NetRexx methods themselves do NOT have a terminating `end` keyword. The `end` keyword is strictly used to close structured blocks like `do`, `loop`, or `select`. Never place an `end` at the end of a method body unless it corresponds to a matching block opener.
8. Exception Signaling Constraints:
   You can ONLY catch checked exceptions (like `SQLException` or `Exception`) if a method that throws/signals them is actually called inside the corresponding `do` block. If a method does not execute any throwing database operations (e.g., the command line `main` driver, or basic helper methods), do NOT use a `do/catch/finally` block. Simply run the statements directly without try-catch blocks.
